🔍 Diagnosi

Dai log Cloud Run è emerso lo stack trace reale:

Error: Value for argument "data" is not a valid Firestore document.
Cannot use "undefined" as a Firestore value (found in field "intent.steps").
If you want to ignore undefined values, enable `ignoreUndefinedProperties`.
    at validateUserInput .../firestore/build/src/serializer.js:383
    at WriteBatch.set .../firestore/build/src/write-batch.js:263
    ...

Causa: parseAndValidateIntent() in handlers/nexus.js ritorna un oggetto con steps: undefined e reasoning: undefined quando quelle chiavi non sono presenti nella risposta di Haiku. Questo oggetto viene poi passato direttamente a writeNexusMessage() che lo serializza per Firestore, dove i campi undefined sono vietati per default.

Il crash era stato introdotto dai miei ultimi cambi al flusso NEXUS (aggiunta training intercept + routing orchestrator). Prima dell'ultimo deploy, i campi opzionali venivano omessi o il path non li includeva.

🔧 Fix applicato

In handlers/shared.js, subito dopo getFirestore():

try { db.settings({ ignoreUndefinedProperties: true }); } catch {}

Questo setting vale per tutte le write sulla Firestore primaria (nexo-hub-15f2d) e previene crash analoghi in futuro per qualsiasi handler. È una best-practice sugerita dalla stessa libreria nel messaggio d'errore.

🔕 FCM disabilitato temporaneamente

Gli errori 401 da FCM erano causati da VAPID key placeholder in index.html. Per evitare rumore nei log della PWA ho commentato l'invocazione di setupFcmPush():

// setTimeout(() => { setupFcmPush().catch(...); }, 1500);

Commento lasciato con istruzioni per riattivare: VAPID key va presa da Firebase Console → Project settings → Cloud Messaging → Web Push certificates. Tutto il resto del codice FCM (Service Worker, handler backend sendPushNotification(), endpoint /nexoPushSend) resta deployato e pronto all'uso.
