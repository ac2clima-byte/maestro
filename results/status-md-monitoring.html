STATUS.md su GitHub — monitoring Claude Chat

Ogni cambio fase di MAESTRO ora scrive STATUS.md nella root del repo e lo pusha. Claude Chat può fare git pull e leggere lo stato senza dipendere da Firestore.

~5

0

Logica di deduplica

CasoComportamento
fase di lavoro nuova (found_task, sending_to_code, waiting_result, pushing_result)PUSH
stessa fase con task diverso (found_task A → found_task B)PUSH
stessa fase di lavoro con stesso taskSKIP
idle ripetutoSKIP
polling ripetutoSKIP
transizione fra fasi rumorose (polling↔idle)SKIP

Test logica (3 scenari simulati)

Scenario 1 — 1 task completo + 3 cicli idle: 6 push (5 fasi + wrap polling)
Scenario 2 — 100 cicli alternati polling/idle senza task: 1 push
Scenario 3 — 2 task back-to-back: 10 push (5 per task)

Cosa scrive STATUS.md

# Claude Code Status
Fase: waiting_result
Task: nome-task
Dettagli: Claude Code sta lavorando
Timestamp: 2026-04-26T09:30:00.000Z
Uptime: 1234s

Come legge Claude Chat

git pull origin main
cat STATUS.md

Implementazione

  • maestro.mjs · updateStatus(fase, dettagli)
  • Firestore (nexo_code_status/current): write sempre — utile per la dashboard.
  • STATUS.md + git push: solo se la transizione cambia stato significativo.
  • Commit mirato (git commit STATUS.md) per non trascinare result/task pendenti.
  • Errori git silenziati: il loop MAESTRO non si blocca per problemi di push.

Stato corrente

Il file STATUS.md è stato creato manualmente con valori bootstrap. L'aggiornamento dinamico parte solo dopo restart MAESTRO (il processo PID 30594 attuale ha ancora il codice precedente).
